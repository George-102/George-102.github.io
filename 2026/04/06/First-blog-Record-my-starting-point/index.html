First blog--Record my starting point

本文最后更新于 2026-04-06T22:16:53+08:00

在这篇博客里，我想和大家分享一下近期关于编程学习与个人生活的感悟

Seven Suggestions for Code

  • 不必追求“全知全能” 以前学习时，我总喜欢钻研极偏的问题（比如 C++ 中 nullptr 与 NULL 的细微底层差异）。虽然钻研能带来新启发，但往往“过犹不及”。我们应学会适时放弃，尝试用 50% 的时间换取 80% 的核心成果，而非耗费 90% 的精力去死磕那 1% 的边缘细节。在这个 AI 触手可及的时代，人的本性是遗忘，学会利用 AI 作为辅助大脑，比记住每一个细节更重要。

  • 1:4 原则：教程一小时，实践四小时 跟着好老师听课时，我们常产生“我学会了”的错觉。但请记住：“听懂”和“做出来”是两码事。 不要只是机械地照搬教程步骤，脱离视频，独立动手实现一遍，才是真正的内化。

  • 完美主义是进步的枷锁 事事追求完美往往会消耗大量的行动力。如果你在学 Python，不要等看完一整本书才写代码。在每个阶段尝试实现一个小功能，比追求“学完再开始”更有意义。

  • 没有所谓的“准备好了”，现在就出发 扎克伯格曾说：你永远无法等一切都准备完美再行动。世界上绝大多数伟大的尝试，都是在动手过程中不断解决问题、收获知识。如果你想建一个网站，不要纠结语言和技术栈，用你最熟悉的工具，先搭起框架，再一步步填充细节。

  • 真正的核心竞争力是解决问题的能力 我们没必要非得花数年时间去“咬透”《C Primer Plus》，除非你的目标是成为语言专家。学习需要变通，不要死板地背诵知识点，要把知识转化为你的“工具库”，学会如何用它去搞定具体的需求。

  • 用户关心的是功能，而非代码的华丽 无论你是用最基础的 cin/cout 堆砌，还是用高深的抽象类封装，别人首先关注的是代码能否跑通、功能是否实现。虽然软件工程的规范非常重要，但在达到那个高度之前，你首先要证明自己具备解决问题的逻辑。

  • 倦怠感真实存在，请保护好你的精力 前段时间我曾陷入自我加压的怪圈：嵌入式、操作系统、数字设计、Kotlin……过载的任务让我的多巴胺耗尽，状态急剧下滑。物极必反，学得多并不代表学得好。重视精力管理，不要让挫败感透支你的热情。

    最后送给大家一句话：我们一直寻找的，却是自己原本早已拥有的；我们总是东张西望，唯独漏了自己想要的，这就是我们至今难以如愿以偿的原因。
